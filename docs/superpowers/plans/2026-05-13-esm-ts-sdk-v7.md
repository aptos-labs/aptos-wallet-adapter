# ESM-only + Aptos TS SDK v7 + Subpath Imports — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move every package to `@aptos-labs/ts-sdk@^7.0.0`, refactor imports to SDK subpath exports where they exist (for cleaner deps and faster type resolution; bundle-size win is already captured by the SDK's `sideEffects: false`), and ship the breaking peer-dep bump with changesets.

**Architecture:** Three sequential commits — (1) version bump + build/test green, (2) subpath import refactor, (3) changesets — on the existing `esm-only` branch. No structural changes to the build system, exports map, or package boundaries.

**Tech Stack:** pnpm 10.33, Turbo, Vite library builds, Vitest, Biome, TypeScript 6.0.2, changesets, `@aptos-labs/ts-sdk@7.0.0`.

## Subpath routing rules

| Symbol(s) | v7 import path |
|---|---|
| `Hex`, `HexInput`, `Signature`, `AccountPublicKey`, `Ed25519Signature`, `Ed25519PublicKey`, `Serializable`, `AbstractedAccount`, `AccountAuthenticatorAbstraction` | `@aptos-labs/ts-sdk/crypto` |
| `Serializer`, `Deserializer` | `@aptos-labs/ts-sdk/bcs` |
| `AptosConfig` (when imported alone), transaction-submission helpers | `@aptos-labs/ts-sdk/transaction` |
| `Network`, `NetworkToChainId`, `NetworkToNodeAPI`, `Aptos`, `AccountAddress`, `AccountAddressInput`, `AuthenticationKey`, `RawTransaction`, `SimpleTransaction`, `MultiAgentRawTransaction`, `MultiAgentTransaction`, `FeePayerRawTransaction`, `RawTransactionWithData`, `TransactionPayloadEntryFunction`, `AnyRawTransaction`, `hashValues`, `generateSigningMessageForTransaction`, `RAW_TRANSACTION_SALT`, `RAW_TRANSACTION_WITH_DATA_SALT`, `isValidFunctionInfo`, `sleep`, `Account` | **stay at root** `@aptos-labs/ts-sdk` |

> Per-file split rule: if a file imports symbols from both crypto+root buckets, emit two `import` statements. Keep `type` and value imports correctly distinguished — never collapse a `type X` into the value namespace.
